# podman-lab — Skill Definition

## Task Scope

Use this skill when working with the `podman-lab` repository at
`~/Documents/code/podman_lab/`. It defines patterns, conventions, and
architecture for a fully self-hosted software lifecycle lab using Podman
Quadlets as systemd services.

## Standard Deployment Workflow

Follow this sequence when deploying or re-deploying the lab:

```bash
# 1. Deploy configs + sync secrets
./setup.sh

# 2. Start all services (dependency order)
./setup.sh --start

# 3. Start cloudflared tunnel (not managed by setup.sh)
systemctl --user start cloudflared

# 4. Verify external access
curl -s -o /dev/null -w "%{http_code}" https://git.runemal.cloud
curl -s -o /dev/null -w "%{http_code}" https://ci.runemal.cloud

# 5. Create Gitea admin + OAuth2 app (manual)
#    Visit https://git.runemal.cloud, create user
#    Settings → Applications → OAuth2 → Register Woodpecker CI
#    Redirect URI: https://ci.runemal.cloud/authorize

# 6. Update secrets.env with OAuth credentials
#    Edit ~/.local/share/podman-lab/secrets.env
#    Set WOODPECKER_GITEA_CLIENT and WOODPECKER_GITEA_SECRET
#    systemctl --user restart woodpecker-server

# 7. Create language repos with CI pipelines
./create-repos.sh --token <GITEA_API_TOKEN>

# 8. Enable Woodpecker for each repo in Gitea settings
```

## Key Conventions

### Podman preference
This lab runs entirely on **podman**. Follow these conventions:
- Use `Containerfile`, never `Dockerfile` — both work with podman, but
  `Containerfile` signals intent and avoids Docker assumptions
- Use the [Woodpecker Podman plugin](https://woodpecker-ci.org/plugins/podman)
  (`head1328/woodpecker-ci-plugin-podman`) for CI image builds — it runs
  `podman build` + `podman push` natively
- Use full image names in Containerfiles (`docker.io/library/python:3.12-slim`)
  to avoid short-name resolution errors in non-interactive podman
- Prefer `podman` CLI commands in scripts and troubleshooting
- Quadlet files use podman-native options (`PodmanArgs`, `Volume`, `Network`)

### Placeholder pattern
- `__USER__` → replaced with `$USER` by `setup.sh` (non-path uses only)
- `__UID__` → replaced with `$(id -u)` by `setup.sh`
- `__DOMAIN__` → replaced with domain (default `runemal.cloud`)
- `__POSTGRES_IP__`, `__REDIS_IP__`, `__GITEA_IP__`, `__WOODPECKER_IP__`, `__MINIO_IP__`, `__REGISTRY_IP__` → replaced with static IPs (default 10.89.1.x)
- Host volume paths use `%h` (systemd `$HOME` specifier), NOT `__USER__`
- NEVER hardcode usernames, UIDs, or IPs in quadlet files — always use placeholders

### Path layout
```
~/.local/share/podman-lab/
├── postgres/              # DB data + init.d/
├── redis/
├── gitea/                 # Repos, LFS, config
├── woodpecker/
├── registry/
├── minio/
├── backups/               # 30-day rotating backups
├── secrets.env            # chmod 600, NEVER commit
└── woodpecker-pipeline.env  # chmod 600, injected into CI steps
```

### Secrets management
- Generated by `setup.sh` via `openssl rand -hex N` on first run
- Stored in `~/.local/share/podman-lab/secrets.env`
- Referenced in quadlets via `EnvironmentFile=%h/.local/share/podman-lab/secrets.env`
- **All secrets are preserved across `setup.sh` re-runs** (passwords, tokens, OAuth credentials)
- Only generates new values when `secrets.env` doesn't exist yet
- To force regeneration: `rm ~/.local/share/podman-lab/secrets.env && ./setup.sh`
- NEVER commit secrets to the repo

### CI Pipeline secrets
- `~/.local/share/podman-lab/woodpecker-pipeline.env` contains MinIO credentials
  and registry URL — injected into every Woodpecker pipeline step via
  `WOODPECKER_ENV_FILE`
- Mounted read-only into the `woodpecker-server` container
- Separate from `secrets.env` to avoid exposing database passwords to pipelines
- MinIO credentials can be overridden: `MINIO_ACCESS_KEY=x MINIO_SECRET_KEY=y ./setup.sh`
- Endpoint URLs (registry, MinIO) are hardcoded in `.woodpecker.yml` files since
  they're infrastructure config, not secrets

### PostgreSQL password sync
On **fresh installs** (no `secrets.env`), `setup.sh` generates new secrets and
syncs PostgreSQL passwords. On re-runs, all secrets are preserved so no sync
is needed.

If you force-regenerate secrets (deleted `secrets.env`), the sync runs
automatically:

```bash
# Only runs when secrets were freshly generated:
podman exec -i postgres psql -U postgres <<SYNCEOF
ALTER USER gitea WITH PASSWORD '${GITEA_DB_PASS}';
ALTER USER woodpecker WITH PASSWORD '${WOODPECKER_DB_PASS}';
SYNCEOF
systemctl --user restart gitea woodpecker-server
```

### Quadlet patterns

**Container with DB dep (static IP + `/etc/hosts` injection):**
```ini
[Unit]
Description=...

[Container]
Image=docker.io/org/image:tag
ContainerName=svc-name
Network=lab-net.network
IP=__SVC_IP__                        # static IP placeholder
PodmanArgs=--add-host=postgres:__POSTGRES_IP__ --add-host=redis:__REDIS_IP__
PublishPort=3000:3000
Volume=%h/.local/share/podman-lab/svc:/data:Z
EnvironmentFile=%h/.local/share/podman-lab/secrets.env
Environment=KEY=static-value

[Service]
Restart=always

[Install]
WantedBy=default.target
```

> ⚠️ podman 4.9 (CNI backend) lacks container DNS resolution. Always use
> `IP=` for infrastructure services and `PodmanArgs=--add-host=` for
> consumers that reference them by hostname.

**Volume:**
```ini
[Volume]
```

**Network:**
```ini
[Network]
```

**Dev container (sleeps until attached):**
```ini
[Container]
Image=docker.io/library/lang:tag
ContainerName=lab-lang
Network=lab-net.network
PublishPort=8080:8080           # use unique host ports per container
Volume=%h/Projects:/workspace:Z
Exec=sleep infinity

[Service]
Restart=always
SuccessExitStatus=0 143
```
> Dev containers use shifted host ports to avoid conflicts with infrastructure:
> `lab-node:3001/5174`, `lab-python:5001/8001`, `lab-go:8081`, `lab-java:8082`, `lab-rust:8083`.

**Oneshot backup container:**
```ini
[Container]
Image=docker.io/alpine:latest       # installs postgresql-client at runtime
ContainerName=lab-backup
Network=lab-net.network
Volume=...:/backup:Z
Volume=...:/data/gitea:Z,ro
Volume=...:/data/registry:Z,ro
EnvironmentFile=...
Exec=/bin/sh -c "apk add --no-cache postgresql-client && pg_dumpall ... && tar ... && find ... -delete"
[Service]
Type=oneshot
```

### Woodpecker agent pattern
Each agent is identical except `ContainerName` and `WOODPECKER_AGENT_LABELS`:
```ini
[Container]
Image=docker.io/woodpeckerci/woodpecker-agent:v3
ContainerName=wp-agent-<lang>
Network=lab-net.network
PodmanArgs=--add-host=woodpecker-server:__WOODPECKER_IP__
Volume=/run/user/__UID__/podman/podman.sock:/var/run/docker.sock:Z
Environment=WOODPECKER_SERVER=woodpecker-server:9000
Environment=WOODPECKER_MAX_WORKFLOWS=1
Environment=WOODPECKER_AGENT_LABELS=type=<lang>
EnvironmentFile=%h/.local/share/podman-lab/secrets.env
```

Available labels: `python`, `node`, `go`, `rust`, `java`.

### Gitea + PostgreSQL config
- Gitea connects via `GITEA__database__*` env vars from secrets.env
- `GITEA__server__LFS_START_SERVER=true` enables Git LFS
- `GITEA__security__INSTALL_LOCK=true` skips the install page (first user to register becomes admin)
- Session and cache use Redis provider
- `WOODPECKER_OPEN=true` must be set for user registration
- Init SQL in `postgres/init.d/init.sql` (generated by `setup.sh` with passwords) creates users + databases at first launch

### Networking
- `lab-net.network` = internal bridge for inter-container communication
- Only Gitea, Woodpecker, Registry, MinIO publish ports to host
- Postgres, Redis, agents, dev containers are lab-net only
- Cloudflared (host network) routes `*.runemal.cloud` → localhost:PORT
- Cloudflared is a separate quadlet (`~/.config/containers/systemd/cloudflared.container`), NOT managed by `setup.sh`
- **Container DNS**: podman 4.9 (CNI backend) does not provide DNS for
  container names. Each infrastructure service gets a **fixed IP**:
  - `postgres` → `10.89.1.2` (`__POSTGRES_IP__`)
  - `redis` → `10.89.1.3` (`__REDIS_IP__`)
  - `gitea` → `10.89.1.4` (`__GITEA_IP__`)
  - `woodpecker-server` → `10.89.1.5` (`__WOODPECKER_IP__`)
  - `minio` → `10.89.1.6` (`__MINIO_IP__`)
  - `registry` → `10.89.1.7` (`__REGISTRY_IP__`)
- Containers that reference others by hostname inject `/etc/hosts` entries
  via `PodmanArgs=--add-host=` — see the IP assignment table in README.md

### Stale CNI leases
If a container fails with `requested IP address X.X.X.X is not available`,
a stale CNI lease exists. Fix:

```bash
rm -f /run/user/$(id -u)/libpod/tmp/rootless-netns/var/lib/cni/networks/systemd-lab-net/<IP>
systemctl --user restart <service>
```

Nuclear option (wipes all containers/volumes/networks):
```bash
podman system reset --force
./setup.sh && ./setup.sh --start
```

### Language repos with CI
`create-repos.sh` creates Git repos on Gitea with boilerplate source files
and `.woodpecker.yml` CI pipelines:

```bash
./create-repos.sh --token <TOKEN>                              # uses git.runemal.cloud
./create-repos.sh --token <TOKEN> --git-url https://my.gitea.com  # custom URL
```

Creates 5 repos: `lab-python`, `lab-node`, `lab-go`, `lab-rust`, `lab-java`.
Each has source files, tests, and a CI pipeline targeting the matching agent.

### Container image builds
Use the [Woodpecker Podman plugin](https://woodpecker-ci.org/plugins/podman)
to build and push OCI images from `Containerfile`:

```yaml
  - name: container-build
    image: head1328/woodpecker-ci-plugin-podman
    privileged: true
    settings:
      repo: 10.89.1.7:5000/my-app
      tags:
        - "${CI_COMMIT_SHA:0:8}"
        - latest
      registry: 10.89.1.7:5000
      tls_verify: false
      containerfile: Containerfile
```

- Runs `podman build` + `podman push` inside the step
- Requires `privileged: true` and trusted repo setting
- Use `Containerfile` (not `Dockerfile`) for podman-native builds
- `registry` field has no `http://` prefix
- Tags support shell expansion: `"${CI_COMMIT_SHA:0:8}"`

### Backup strategy
- Daily `pg_dumpall` for PostgreSQL
- Daily `tar -czf` for Gitea data and Registry blobs
- Stored in `backups/` volume with 30-day retention via `find -mtime +30 -delete`
- Triggered by `lab-backup.timer` at 03:00 daily
- Backup container uses `alpine:latest` image (installs `postgresql-client` at runtime)

## Adding a new service

1. Create `quadlets/<name>.container` using the patterns above
2. Add data directory to setup.sh (mkdir line)
3. If external access needed: add `PublishPort` + Cloudflare tunnel entry
4. If DB needed: add `CREATE USER` and `CREATE DATABASE` to `config/postgres/init.sql` (template); `setup.sh` generates the deployed copy at `postgres/init.d/init.sql` with passwords injected
5. If CI agent: copy an existing `wp-agent-` file, change label + name
6. **IP management**: if other containers resolve this service by hostname:
   - Add a `__SVC_IP__` placeholder to the new container via `IP=`
   - Register it in setup.sh with a default IP and sed substitution
   - Add `PodmanArgs=--add-host=svc:__SVC_IP__` to every container that
     references the new service by hostname

## Verification commands

```bash
# Service status
./setup.sh --status

# Individual service check
systemctl --user status <service>
journalctl --user -u <service> --no-pager | tail -30

# Connectivity
curl -s -o /dev/null -w "%{http_code}" http://localhost:3000   # gitea
curl -s -o /dev/null -w "%{http_code}" http://localhost:8000   # woodpecker
curl -s -o /dev/null -w "%{http_code}" http://localhost:5000/v2/  # registry
curl -s -o /dev/null -w "%{http_code}" http://localhost:9000/minio/health/live  # minio

# Podman state
podman ps --all
podman network ls
podman volume ls

# Cloudflared
systemctl --user status cloudflared
journalctl --user -u cloudflared --no-pager | tail -20
```

## Troubleshooting

| Symptom | Fix |
|---------|-----|
| `requested IP address not available` | Stale CNI lease — remove file or `podman system reset --force` + redeploy |
| `password authentication failed` | Secrets are preserved; shouldn't happen. Fix: `podman exec postgres psql -U postgres -c "ALTER USER x WITH PASSWORD 'y';"` + update `secrets.env` |
| `Client ID not registered` | OAuth creds preserved; if missing, paste into `secrets.env` + restart woodpecker-server |
| `registration is closed` | Set `WOODPECKER_OPEN=true` in `woodpecker-server.container` |
| Cloudflared not routing | `systemctl --user start cloudflared` |
| Container won't start | `journalctl --user -u <service>` |
| Port conflict | `ss -tlnp \| grep <PORT>` |
| Agent won't connect | Verify gRPC port 9000, check `WOODPECKER_AGENT_SECRET` matches |
| Gitea can't reach Postgres | Both must be on `lab-net`; check `podman exec gitea cat /etc/hosts` |
| DNS lookup fails | podman 4.9 lacks container DNS — uses static IPs + `/etc/hosts` |
| Volume permission denied | Add `:Z` to volume mounts |
| Podman socket error | Ensure `/run/user/$UID/podman/podman.sock` exists |
